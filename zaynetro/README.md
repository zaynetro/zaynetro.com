# zaynetro.com

Personal website and blog built with [Phoenix](https://phoenixframework.org/) and
[Phoenix LiveView](https://hexdocs.pm/phoenix_live_view). Posts are written in Markdown
with TOML frontmatter and rendered server-side using [MDEx](https://github.com/leandrocp/mdex)
(comrak + Autumn syntax highlighting). Images are processed on-demand via the
[`image`](https://hex.pm/packages/image) NIF library (libvips is bundled; no system install required).

## Running locally

```sh
# Install Elixir dependencies and build JS/CSS assets
mix setup

# Start the dev server (with live-reload and asset watchers)
mix phx.server
# — or inside IEx —
iex -S mix phx.server
```

The site is available at **http://localhost:4000**.

The Phoenix LiveDashboard (metrics, processes, ETS tables, etc.) is available at
**http://localhost:4000/dev/dashboard** in the `dev` environment.

## Running remotely

Need to run these commands once.

```sh
mix phx.gen.release 
mix phx.gen.secret
fly secrets set SECRET_KEY_BASE=<output of mix phx.gen.secret>
```

- See more at <https://hexdocs.pm/phoenix/fly.html>

## Pages

### Home — `/`

Lists all published blog posts sorted by date (newest first). In development, draft posts
are visible; in production they are hidden (`hide_drafts: true`).

### About — `/about`

Static page rendered from `lib/zaynetro_web/controllers/page_html/about.html.heex`.

### Blog post — `/post/:slug`

Each post lives under `posts/` either as:

- `posts/{slug}/index.md` — directory-based post (can contain co-located images)
- `posts/{slug}.md` — single-file post

**Frontmatter** is TOML delimited by `+++` lines. Required fields: `title`, `date`.
Optional: `description`, `draft`, `[extra.preview_image]`.

The rendering pipeline:

1. **Startup** — `Zaynetro.Blog.Cache` (GenServer) calls `Blog.Loader.load_all/1`, which
   discovers every post file, parses frontmatter, and renders Markdown via `Blog.Renderer`.
2. **Renderer** — a single MDEx AST traversal:
   - Extracts H2/H3 headings into a nested table-of-contents (TOC).
   - Converts local `![alt](./img.png)` images into `<picture>` elements with a
     `srcset` pointing at the `/img` pipeline (multiple widths: 400 – 1200 px).
   - Transforms `<mermaid-block>` custom tags into Mermaid diagram markup.
   - Transforms `<labeled-img>` custom tags into captioned image sections.
   - Detects `has_mermaid` / `has_labeled_img` flags so the template can conditionally
     load the Mermaid JS library.
3. **Disk cache** — rendered HTML is written to `tmp/html/{slug}.html` so subsequent
   server restarts skip the MDEx render step.
4. **ETS cache** — all posts and image paths are held in ETS tables
   (`:blog_posts`, `:blog_images`) for lock-free concurrent reads.

### Explainix — `/explainix`

Interactive visual explainer for Nix language syntax. The page shell is server-rendered
HEEx; the interactive widget is a **Preact island** (`assets/js/islands/explainix.jsx`)
loaded via a deferred `<script>` tag (`/assets/explainix.js`). Hover over any element to
highlight its bounds; click to show contextual help.

### Sudoku — `/sudoku` and `/sudoku/:exercise`

Interactive Sudoku technique demonstrations. The index page lists available exercises;
each exercise page is a **Phoenix LiveView** (`ZaynetroWeb.SudokuLive`) that mounts the
exercise name into a `data-exercise` attribute and delegates all rendering to the Preact
island (`assets/js/islands/sudoku.jsx`, bundled as `/assets/sudoku.js`).

Available exercises: `x-wing`, `y-wing`, `empty-rectangle`.

### RSS feed — `/feed.rss`

Atom/RSS feed of published posts, generated by `FeedController`.

### Sitemap — `/sitemap.xml`

XML sitemap generated by `SitemapController`.

## Image pipeline — `/img`

All images (both post images and static images) are served through a single endpoint:

```
GET /img?id=<image-id>&w=<width>          # resize to width, returns PNG
GET /img?id=<image-id>&orig=true          # serve original file (any format)
```

An optional `v=<hash>` query parameter enables permanent HTTP caching
(`Cache-Control: public, max-age=31536000, immutable`).

The pipeline (`Zaynetro.Images.Pipeline`):

1. **Lookup** — resolve the image ID to an absolute file path via the ETS images table.
   Post images are registered as `{slug}/{filename}`; static images from
   `priv/static/images/` are registered by filename.
2. **Disk cache check** — look for a pre-resized file at
   `{image_cache_dir}/{id_dir}/{basename}_{w}w.png`.
3. **Resize** — if not cached, enqueue a libvips thumbnail job through
   `Zaynetro.Images.Queue`. The queue is a single-slot GenServer that serialises all
   libvips calls to prevent resource exhaustion.
4. **Serve** — stream the file to the client with the appropriate `Content-Type`.

The image cache directory defaults to `tmp/image_cache` in dev and
`$IMAGE_CACHE_DIR` (or `$TMPDIR/zaynetro/images`) in production.

## Configuration

| Config key | Dev default | Production env var |
|---|---|---|
| `posts_dir` | `posts/` (relative to project root) | `POSTS_DIR` |
| `image_cache_dir` | `tmp/image_cache/` | `IMAGE_CACHE_DIR` |
| `hide_drafts` | `false` | hardcoded `true` in prod config |
| Endpoint secret | hardcoded dev value | `SECRET_KEY_BASE` |
| Host | `localhost` | `PHX_HOST` |
| Port | `4000` | `PORT` |

## Asset bundles

Three separate esbuild entry points are compiled:

| Bundle | Entry point | Loaded on |
|---|---|---|
| `zaynetro` | `assets/js/app.js` | every page |
| `sudoku` | `assets/js/islands/sudoku.jsx` | `/sudoku/:exercise` |
| `explainix` | `assets/js/islands/explainix.jsx` | `/explainix` |

Tailwind CSS is compiled by the `tailwind` hex package watcher.
